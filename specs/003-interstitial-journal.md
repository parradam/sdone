# 003: Interstitial journal

**Status:** Draft — ready to implement ([004 app-shell implementation](004-app-shell-implementation.md) is done)
**Owner:** Adam

## Goal

### Backend

- create a model for the interstitial notes, including id, submitted time, note
- perform the relevant migrations
- the ability to fetch more notes should be built in

### Frontend

- create a tabular view with a timestamp that is autogenerated when submitted; until then, it should show the current time, which becomes final on pressing return
- make distinctive the characters todo regardless of case, assuming they are not part of a longer word; it should be highlighted if followed by a colon or space, for example
- notes should be in chronological order
- notes should be scrollable
- fetch more notes when scrolling upwards
- allow editing of previous notes and mark it as "edited"

## Context

  the adopted "cool slate" direction from spec 002. The journal lives on the

- This is the **first feature page**. It depends on **spec 004 (app shell)**,
  which establishes the shared `base.html`, the `StaticFiles` mount, HTMX, and
  the Tailwind build. Implementation of 003 is blocked until 004 is done; this
  spec assumes those exist and only adds journal-specific templates/partials.
- Reuse the backend foundations from **spec 001**: the `get_db` session
  dependency, the `Base` declarative model + Alembic setup, the shared error
  handling, and the transactional test fixtures (`client`, `db_session`) in
  `backend/tests/conftest.py`.
- No authentication or users exist yet (out of scope in 001). Notes are therefore
  single-user / global for now; a `user_id` foreign key will be added when the
  auth spec lands.

## User-facing behaviour

- As a user, I can record and view interstitial notes so that I can stay on task and track my progress through a study session
- Allow scrolling of notes
- Show a sticky day/date for the relevant day's notes
- If the notes go over a day, show the new day/date
- Indicate focus of the current note
- If there are no notes to load, show only an option to enter a new note
- On error, show a polite message and do not allow entry of notes

## Scope

### In scope

- **Model** `InterstitialNote` + an Alembic migration.
- A **`journal` APIRouter** (the first router in the app), included in `main.py`.
- **HTMX HTML-fragment endpoints** (full page + row partials), not a JSON API.
- **Journal page + partials** extending the shell `base.html` from spec 004.
- A server-side **`highlight_todo` Jinja filter** (standalone `todo` keyword).
- **Composer** with a live "current time" clock that finalises on submit.
- **Reverse infinite scroll**: load older notes when scrolling up (keyset paging).
- **Inline edit** of an existing note, marking it `edited`.
- **Day grouping** with sticky per-day headers and dividers across day boundaries.
- **Empty state** (composer only) and **error state** (polite message, composer
  disabled).
- Unit + integration tests (see Technical approach).

### Out of scope

- The app shell, `base.html`, `StaticFiles` mount, HTMX inclusion, and the
  Tailwind build — all delivered by **spec 004**.
- Authentication, users, and per-user note scoping (`user_id`).
- Linking notes to tasks or sessions; the "Just noted" dashboard panel.
- Search over notes; deleting notes; markdown / rich text; tags beyond the
  presentational `todo` highlight.
- Realtime updates / multi-device sync.

## Technical approach

**Testing (TDD, per CLAUDE.md).** Write the failing test first for each unit,
then the code, then refactor. Reuse the spec 001 harness (in-memory SQLite,
transactional `db_session`, `TestClient`-based `client`). Commit atomically and
only when `ruff check`, `mypy .`, and `pytest` all pass. Test-first order:

1. **Model** — `tests/test_note_model.py`: `created_at` is set and timezone-aware;
   `edited_at` defaults to `None`; body is trimmed; empty/whitespace body is
   rejected; body over the max length is rejected.
2. **`highlight_todo` filter** — `tests/test_highlight_todo.py`, table-driven:
   `"todo: x"` and `"TODO x"` wrap the keyword; `"todos"` / `"todolist"` do **not**
   match (longer word); a standalone `"... todo ..."` mid-sentence matches; the
   note body is **HTML-escaped** so a body like `<script>` cannot inject markup
   (the filter escapes first, then wraps the keyword, and returns markup-safe
   output).
3. **Endpoints** — `tests/test_journal_api.py` via `TestClient`, one test per
   behaviour: the page renders the composer; `POST` creates a note and returns the
   new row partial containing the body; an empty/whitespace body returns **422**
   with an inline error and creates no row; keyset pagination (`?before=<id>`)
   returns only older rows in the correct order and respects the page size; `PUT`
   updates the body and marks it `edited` (sets `edited_at`, partial shows the
   marker); the empty state renders the composer only; a DB error renders the
   polite message with the composer disabled (override `get_db` to raise, exactly
   as `tests/test_health.py` does).

**Hard-to-TDD (client-side).** The live composer clock and the scroll-up trigger
are thin JS/HTMX glue. All testable logic (pagination, `todo` highlighting, day
grouping, the `edited` marker, validation) lives **server-side** and is covered by
the tests above. The JS glue (`hx-trigger="revealed"` sentinel, the ticking
clock) is verified manually / via the browser MCP and is called out as such in
the acceptance criteria rather than pretended to be unit-tested.

**Backend.** `models/note.py` defines `InterstitialNote` on `Base`. `routers/journal.py`
is an `APIRouter` holding the four endpoints and using `Depends(get_db)`; DB
access is a few small query helpers (keyset `list before id`, create, update) —
kept in the router or a thin `services` function at Claude's discretion. HTML is
rendered through the `Jinja2Templates` instance from spec 004.

**Frontend.** `templates/journal/index.html` extends `base.html` and fills the
`content` block with the scroll container + composer. Row partials
(`_notes.html`, `_note.html`, `_note_edit.html`) are returned by the fragment
endpoints. Newest notes sit at the bottom; a top sentinel with
`hx-trigger="revealed"` fetches older notes and prepends them with
`hx-swap="afterbegin"`. Minimal JS (`static/js/journal.js`) drives the live clock
and keeps the scroll position stable when older notes are prepended.

## Data model / API contract

**Model** `InterstitialNote` (table `interstitial_notes`):

| column       | type                     | notes                                   |
| ------------ | ------------------------ | --------------------------------------- |
| `id`         | int, PK, autoincrement   | stable keyset cursor                    |
| `body`       | text, not null           | trimmed; non-empty; max ~2000 chars     |
| `created_at` | datetime (UTC), not null | server default `now()`; submit time     |
| `edited_at`  | datetime (UTC), nullable | set on edit; non-null ⇒ show "edited"   |

No `user_id`, `is_todo`, task, or session columns (see Scope › Out).

**Endpoints** (HTMX fragments; all render Jinja partials, not JSON):

```md
GET  /journal
  → 200 full Journal page (extends base.html): the most recent page of notes
    (newest at the bottom) + the composer. Empty DB → composer only.
    DB error → polite message, composer disabled.

GET  /journal/notes?before=<id>&limit=<n>
  → 200 rendered <li> row partials for the page of notes with id < before,
    newest-last, for reverse infinite scroll (prepended above). limit defaults
    to the page size; omit `before` for the first page.

POST /journal/notes            (form: body)
  → 201 the new note row partial (appended at the bottom)
  → 422 the composer partial with an inline validation error (empty/too long);
        no row is created.

PUT  /journal/notes/{id}       (form: body)
  → 200 the updated note row partial, marked "edited" (edited_at set)
  → 404 if the note does not exist
  → 422 the edit partial with an inline error (empty/too long)
```

Server errors surface via the shared envelope / handler from spec 001 where a
JSON response is appropriate; for the interactive endpoints the user sees the
rendered polite-error / disabled-composer state instead.

## Acceptance criteria

- [ ] `InterstitialNote` model + migration exist; `alembic upgrade head` applies
      cleanly. **Unit tests** for defaults (`created_at` set, `edited_at` None),
      body trimming, empty-body rejection, and max-length rejection.
- [ ] `highlight_todo` Jinja filter highlights a standalone `todo`/`TODO`
      followed by a colon or space, ignores it inside a longer word
      (`todos`/`todolist`), and HTML-escapes the body. **Unit tests** (table-driven,
      incl. an XSS/escaping case).
- [ ] `GET /journal` returns 200 HTML with the composer. **Integration test**
      via `TestClient`.
- [ ] `POST /journal/notes` with a valid body creates one note and returns its
      row partial; with an empty/whitespace body returns 422 with an inline error
      and creates no note. **Integration tests.**
- [ ] `GET /journal/notes?before=<id>` returns only older notes, correctly
      ordered, respecting the page size (keyset pagination). **Integration test.**
- [ ] `PUT /journal/notes/{id}` updates the body and marks the note `edited`
      (sets `edited_at`; partial shows the marker). **Integration test.**
- [ ] Empty state renders the composer only; DB-error state renders a polite
      message with entry disabled (`get_db` overridden to raise). **Integration
      tests.**
- [ ] Notes render newest-at-bottom with sticky day headers and a divider when
      the date changes. **Verified manually / via browser MCP.**
- [ ] Composer shows a live current time that finalises on submit; scrolling up
      loads older notes and preserves scroll position. **Verified manually / via
      browser MCP.**
- [ ] `ruff check` and `mypy .` pass on all new code.

## Open questions

- Page size for the initial load and each scroll-up fetch (proposed: 50).
- Maximum note body length (proposed: ~2000 chars) — confirm.
- Keyset cursor on `id` alone vs `(created_at, id)`. `id` is simplest and stable;
  `(created_at, id)` only matters if notes can be back-dated (they can't yet).
- Should spec 004 be authored/implemented before 003 work begins? (Yes — 003 is
  blocked on it.)
